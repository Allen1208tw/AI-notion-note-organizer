# 展示腳本與技術問答

## 30 秒介紹

這是一套 AI 文件學習系統。使用者上傳 PDF、Word、TXT 或 Markdown 後，系統會解析文字、辨識主章節，再用 OpenAI 產生結構化學習筆記。內容可匯出到 Notion，Quiz 和 Flash Card 會存入 SQLite，後續可練習、追蹤弱點和查看儀表板。系統也有分層快取、Notion 續跑和資料診斷，避免失敗就全部重做。

## 五分鐘展示順序

1. 雙擊安裝版捷徑，說明 Launcher 會檢查 Schema、選擇 Port，並同時啟動 Streamlit 與背景 Worker。
2. 上傳文件，展示格式驗證、章節描述性標題和分析結果。
3. 說明 Chunk 分析與逐章詳細筆記是兩條互補流程。
4. 開啟 Notion，展示父頁、章節子頁、Callout、學習地圖、Quiz/Flash Card Toggle。
5. 進 Quiz，作答並自評，展示 Attempt 與 WeakPoint。
6. 進 Flash Card 翻面評分，說明 Review 和排程。
7. 開儀表板，指出得分率、熟悉度和弱點。
8. 開資料診斷，展示每章分布、快取同步和去重能力。
9. 開背景工作與更新頁，說明耗時工作可跨頁執行，更新檔會驗證 SHA-256。

## 架構說法

「我採分層式單體。Streamlit 只管互動，Parser 和 Processor 管文件，Service 管業務規則與外部整合，Pydantic 管 AI 資料契約，SQLAlchemy/SQLite 管不可替代的學習紀錄。這樣目前維持單機簡單部署，未來也能把 Service 放到 FastAPI 後面。」

## 常見問答

### 為何使用 Chunk

長文件可能超出上下文，且一次分析容易忽略後段。Chunk 加 overlap 保留邊界語意，再由 Merge Prompt 整合全局結果。

### 如何保證 AI 回傳格式

Prompt 明確要求 JSON，Service 解析並用 Pydantic 驗證；失敗有限次重試。Pydantic 保證結構，不宣稱保證內容真實。

### 如何避免重複花 API 費用

PDF 視覺分析和詳細筆記分開快取。Notion 失敗可用已完成的 AI 快取續跑；舊文件也可直接把快取題目回填 SQLite。

### 為何快取不用 SQLite

AI 中間產物體積大且可重建，檔案快取更直接；作答與複習不可重建，所以放關聯式資料庫。這是資料價值與生命週期的區分。

### 如何避免重新同步刪掉練習歷史

題目同步採 merge，使用正規化問題+答案或正面+背面作 identity，只新增缺少項目。重新分析也沿用匹配章節 ID。

### 為何 WeakPoint 不是每次答錯新增一筆

Attempt 保存事件歷史，WeakPoint 保存目前摘要。每題一筆 WeakPoint 可累積錯誤次數和分數，避免錯題本出現大量重複卡片。

### SQLite 有什麼限制

適合單機、零部署，但不適合多人高併發；`create_all` 也不會修改舊表。雲端版會改 PostgreSQL 並導入 Alembic。

### Notion 和 SQLite 同時失敗怎麼辦

兩者無法共享交易，所以不假裝強一致。系統先保留 AI 快取與章節匯出狀態，Notion 可續跑；SQLite 可由快取單章或整份回填，形成可恢復的最終一致性。

### 章節偵測最難的地方

真實 PDF 會有目錄重複、跨行標題、同名不同章、兩位數被拆字、後段再次出現 Module 序列。偵測器不只用 Regex，還建立目錄映射、選正文主序列、合併標題並保證來源 ID 唯一。

### Streamlit 最常見的坑

每次互動會整頁 rerun，因此持久資料不能只放 Session State；Widget 需要唯一 Key；切頁和切章節時要重設正確狀態。卡片不換張與 Selectbox 空白都曾由這類問題造成。

### 如何測試

除了語法和單元測試，會測重開程式、重複同步、舊快取、部分 Notion 失敗、重複標題、兩位數章節、重新分析是否保留歷史。AI 應用不能只測一次正常輸出。

### 如何雲端化

加入登入與 `user_id`，SQLite 換 PostgreSQL，本機快取換物件儲存，長任務進背景佇列，外層加 API 與權限。現有 Service 可保留，UI 和 Infrastructure 逐步替換。

### 這個專案最大的工程價值

不是單次生成漂亮筆記，而是將不穩定的 AI 和外部 API 包在可驗證、可快取、可續跑、可持久化、可診斷的工作流中。

## 不要過度宣稱

- 不要說 AI 絕對正確；說有結構驗證和來源約束。
- 不要說已是多人 SaaS；目前是單機架構。
- 不要說 Notion 與 SQLite 原子同步；它們採可恢復的最終一致性。
- 不要說已封裝成單一 EXE；目前是 Batch + Launcher 一鍵啟動。
- 不要說 OCR 已完整支援；掃描 PDF 的能力取決於現有解析與視覺流程。

## 收尾句

「我先把文件到學習內容的主流程做通，再逐步補上持久化、練習、弱點、儀表板，最後處理去重、快取回填、章節偵測和診斷。這條路線讓每一版都有可用價值，也讓後期穩定性不是臨時補丁，而是有明確資料模型和恢復策略。」
