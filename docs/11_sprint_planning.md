# Sprint 規劃方法

## 開發單位

每個功能以「資料契約 -> Service -> UI -> 持久化 -> 測試 -> 文件」為完整切片。只做頁面而沒有持久化，不算完成；只做 Model 而沒有舊資料策略，也不算完成。

## 建議工作順序

1. 寫清楚使用者行為和成功條件。
2. 找出會受影響的 Model、Service、Page、快取與外部 API。
3. 先定義輸入/輸出和交易邊界。
4. 實作 Service，再接 UI。
5. 測正常、重跑、重開、部分失敗和舊資料。
6. 更新技術文件與版本紀錄。

## Definition of Done

- 語法檢查通過。
- 相關自動化測試通過。
- Streamlit 操作可完成主要路徑。
- 關閉重開後持久資料仍存在。
- 重複執行不產生重複資料。
- 錯誤訊息包含可採取的下一步。
- 不覆蓋無關的使用者資料或程式修改。
- 文件對應實際檔名與函式。

## 風險導向測試

| 變更 | 必測項目 |
|---|---|
| Model/Schema | 新舊 DB、外鍵、唯一限制、刪除順序 |
| Chapter Detector | 目錄、正文、跨行、兩位數、重複標題、缺章 |
| 快取 | 新格式、舊格式、Hash 不同、缺檔、無效 JSON |
| Quiz/Flash Card 同步 | 首次、重跑、部分已存在、保留歷史 |
| Streamlit 狀態 | 切文件、切章節、rerun、重開、Widget Key |
| Notion | 新建、續跑、部分失敗、舊狀態格式、圖片缺失 |
| 背景工作 | 建立、進度、取消、完成結果、Worker 中止恢復 |
| Windows 發行 | 封裝健康檢查、Worker 心跳、Server health、安裝檔雜湊 |
| 自動更新 | HTTPS、Host allowlist、版本比較、SHA-256 失敗刪檔 |

## Bug 修復模板

```text
現象：使用者看到什麼
根因：資料在哪一層出錯
影響：哪些資料或功能受影響
修正：改動的責任層
相容：舊資料如何處理
驗證：如何證明不復發
```

## V2.5 收尾檢查

- Schema 欄位一致。
- 重新分析保留匹配章節與練習歷史。
- Quiz/Flash Card 同步去重。
- 單章與整份快取回填可用。
- 章節偵測支援描述性和跨行標題。
- 頁面中文化，文件/章節 Selectbox 可操作。
- Quiz 答案可收合，Flash Card 評分後換卡。
- 管理頁可診斷、清除、合併與顯示成功結果。
- `.bat` 與 Launcher 可啟動。
- 備份排除 `.env`、`.venv`、`outputs`。

## 下一個 Sprint 建議

完成文件後，先凍結功能一輪，使用至少三種不同教材做端到端回歸，再選擇 V3 Windows 發行版或 V4 雲端化。不要同時進行打包與多人架構，兩者的資料路徑和部署假設不同。
