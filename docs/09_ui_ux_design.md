# UI 與互動設計

## 資訊架構

側邊欄依使用流程排列：首頁建立內容、文件管理、Quiz、Flash Card、儀表板、資料診斷。名稱使用中文，降低使用者理解成本。

## 首頁

`AI_Notion_筆記整理器.py` 是主要工作台：上傳文件、顯示驗證與分析進度、預覽章節、產生詳細筆記、匯出 Markdown/JSON、建立或續跑 Notion。

耗時操作會顯示進度與目前章節，完成後保留結果到 Session State，避免畫面 rerun 後立即消失。

文件分析與整份 Notion 匯出改由背景佇列執行。首頁顯示目前工作，`pages/6_背景工作.py` 提供跨頁查看、取消與歷史；因此使用者不必停留在同一個 Streamlit 頁面等待。

首頁與背景工作頁使用 Streamlit fragment 自動刷新工作狀態。使用者不需要手動按「更新狀態」；只要瀏覽器分頁仍開著，進度條會定期讀取 SQLite 中的最新進度。若使用者關掉分頁再重開，首頁會重新掃描等待中與執行中的背景工作，避免看起來像整個流程被重置。

## 首次設定

`pages/0_開始使用與設定.py` 讓安裝版使用者直接在網頁輸入 OpenAI Key、Notion Token、父頁和進階參數。Secret 使用 password input，畫面只顯示是否已設定，不回填原值；空白提交會保留既有 Key，清除必須勾選明確選項。

設定保存到 AppData 的 `.env`。由於 Streamlit 與背景 Worker 都會在 import 時載入設定，儲存後由 Launcher 完整重啟兩個程序，而不是只做 `st.rerun()`。首頁偵測不到 OpenAI Key 時會停止分析流程並引導至設定頁。

## 文件與章節選擇器

Quiz、Flash Card、儀表板和管理頁都採兩層選擇：先文件，再章節。選項顯示人類可讀標題，但實際值應直接對應 ID 或索引，不以顯示文字反查，避免相同標題導向錯誤章節。

Widget Key 必須依頁面、文件或模式唯一，否則 Streamlit 會沿用舊狀態或出現 duplicate element ID。

## Quiz 互動

流程是「讀題 -> 輸入答案 -> 展開標準答案 -> 自評」。標準答案按鈕可再次收合，讓使用者先回想，不被答案持續佔據畫面。自評後才寫入資料庫並切換下一題。

## Flash Card 互動

依使用者偏好移除「今日到期」入口，直接以章節練習。卡片先顯示正面，翻面後出現 0 到 5 熟悉度；評分成功後索引加一並 rerun。

索引、是否翻面、卡片 ID 清單都存入章節專屬 Session Key，防止切換章節或其他分頁時互相覆蓋。

## 儀表板

先顯示可掃讀的核心 Metric，再顯示圖表、弱點排行和近期活動。每個數值都應能追溯到 Service 的定義，例如正確率、得分率和完成率不可混用。

## 資料管理

管理頁把技術異常轉成可理解訊息：章節分布不均、重複資料、孤兒紀錄、快取缺失。破壞性操作要求輸入確認文字；成功同步使用 Dialog 顯示文件、章節與新增數量，避免 rerun 後只看到「資料已存在」警告。

## Streamlit rerun 心智模型

每次按鈕、Selectbox 或輸入變動，整個頁面腳本會重新執行。正確做法是：

- 短暫 UI 狀態放 `st.session_state`。
- 需跨重開保留的資料寫 SQLite。
- 每個 Widget 提供穩定且唯一的 `key`。
- 成功訊息需跨 rerun 時，先寫 Session State，再 rerun，下一輪顯示後清除。
- 不讓兩個 Tab 共用會互相重設的索引 Key。

## 錯誤訊息原則

錯誤應包含「發生什麼、資料是否保留、下一步怎麼做」。例如 Notion 單章失敗時，應告知快取仍存在且可續跑，而不只顯示 Python Exception。

## 可及性與文字

按鈕使用清楚動詞，章節標題使用描述性名稱；長文字允許換行，避免用破壞 Selectbox 內部結構的全域 CSS。顏色不是唯一狀態訊號，仍搭配文字和數值。
